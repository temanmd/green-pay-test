# Architecture Decision Records

Краткие записи ключевых решений: **контекст → решение → почему → компромиссы**.
Цель — чтобы любое техническое решение можно было защитить на собеседовании.

---

## ADR-001. Деньги — целые в минимальных единицах (`*_cents`)

**Решение.** Все суммы храним как `integer` в минимальных единицах валюты
(`amount_cents`, `balance_cents`) + отдельный код валюты (`currency`). Никаких
`float`/`double`.

**Почему.** Float не представляет десятичные дроби точно (`0.1 + 0.2 != 0.3`) —
для денег это недопустимо. Целые числа дают точные арифметику и сравнения,
корректно работают в БД-агрегатах (`SUM`) и не накапливают ошибку округления.
Это стандарт индустрии (так делает Stripe).

**Компромиссы.** `decimal(precision, scale)` тоже корректен, но целые проще,
быстрее и однозначнее в JSON. Минус — на границе API нужно помнить про единицы;
решаем это явным контрактом (отдаём `amount_cents` + `currency`).

## ADR-002. Счёт-баланс — append-only леджер + кэш баланса

**Решение.** Движения по счёту — это **неизменяемые** записи в таблице
`transactions` (никогда не `UPDATE`/`DELETE`). `accounts.balance_cents` —
**кэш** (производная величина), который пересчитывается тем же изменением, что
добавляет запись в леджер. Отмена успешного заказа не правит и не удаляет
исходную запись — создаёт **компенсирующую reversal-запись**, ссылающуюся на
оригинал.

**Почему.** Платёжной системе нужен полный неизменяемый аудит-трейл: по леджеру
всегда можно восстановить, как получился баланс, и доказать каждое движение.
Сторнирование через компенсирующую запись (а не «откат») — это то, как работает
настоящая бухгалтерия и платёжные процессоры: история фактов не переписывается.
Кэш баланса нужен для быстрого чтения без `SUM` по всей истории; инвариант
`balance_cents == SUM(transactions.amount_cents)` поддерживается в одной
транзакции.

**Компромиссы.**
- Полный *double-entry* (дебет одного счёта, кредит другого) ещё «правильнее»
  бухгалтерски, но в ТЗ счёт по сути один на юзера — это добавило бы системный
  счёт-контрагент и сущности без пользы для демонстрации подхода.
- Чистый «один столбец `balance` + лог» проще, но теряет строгий аудит и
  смешивает источник истины (баланс) с производной (лог).

## ADR-003. Конкурентность — пессимистичные блокировки (`SELECT … FOR UPDATE`)

**Решение.** Любое изменение баланса идёт внутри `account.with_lock { … }`
(`SELECT … FOR UPDATE`) в одной БД-транзакции вместе с записью в леджер.

**Почему.** Два параллельных запроса на один счёт без блокировки дают
*lost update* (оба прочитали старый баланс, оба записали свой) — деньги
«теряются» или «дублируются». Строчная блокировка сериализует операции по
конкретному счёту, сохраняя параллелизм между разными счетами.
В терминах **CAP**: внутри одной БД-партиции (PostgreSQL, один primary) мы
выбираем **строгую согласованность (C)** — для денег это не обсуждается; цена —
кратковременная сериализация операций по одному счёту.

**Компромиссы.** Оптимистичные блокировки (`lock_version`) дают меньше contention,
но перекладывают ретраи на клиента и хуже подходят для «горячих» счетов с частыми
записями. Под платёжную нагрузку на один счёт пессимистичная блокировка
предсказуемее.

## ADR-004. Идемпотентность — доменная + HTTP `Idempotency-Key`

**Решение.** Два уровня:
1. **Доменный** — переходы статусов через guard'ы state-machine (success только
   из `created`, cancel только из `succeeded`) + **уникальные индексы** в БД
   (одна settlement-запись и одна reversal-запись на заказ).
2. **Транспортный** — заголовок `Idempotency-Key`: повтор запроса с тем же
   ключом возвращает сохранённый первый ответ, а не выполняет операцию заново.

**Почему.** В платежах ретраи неизбежны (таймауты, повторные клики, ретраи
сети). Доменный уровень физически не даёт дважды списать/начислить даже при
гонке — уникальный индекс упадёт на второй вставке внутри транзакции.
Транспортный уровень делает безопасным *весь* POST-запрос целиком (клиент
получает идентичный ответ), что и ожидает платёжная интеграция.

**Компромиссы.** `Idempotency-Key` требует хранилища ключей и аккуратной
обработки «запрос ещё в процессе». Это осознанная цена за корректность —
профильное требование для payments-вакансии.

## ADR-005. CQRS-разделение: команды и запросы

**Решение.** Запись — через command-объекты в `app/services/` (PORO/`mutations`),
чтение — через query-объекты в `app/queries/`. Контроллеры тонкие: парсят вход,
зовут команду/запрос, отдают сериализованный результат.

**Почему.** Команды (меняют состояние) и запросы (только читают) имеют разные
требования: транзакции/локи/инварианты против оптимизации выборок. Разделение
делает бизнес-операции явными, тестируемыми и переиспользуемыми, убирает «жир» из
моделей и контроллеров. Та же структура — в референсном проде.

**Компромиссы.** Чуть больше классов, чем «вся логика в модели». Для платёжной
логики явность важнее краткости.

## ADR-006. Статусы заказа — `aasm`, без AR-колбэков

**Решение.** Статусы и переходы заказа описываем декларативно через `aasm`
(`created → succeeded → cancelled`). Сами события (`succeed!`, `cancel!`)
вызываются **из команд**, а побочные эффекты (проводка по леджеру) живут в
командах, а не в колбэках модели.

**Почему.** Явная state-machine с guard'ами не даёт невалидных переходов
(например, отменить ещё не успешный заказ). Отказ от AR-колбэков (требование ТЗ и
лучшая практика): колбэки прячут побочные эффекты, путают порядок выполнения и
ломают тестируемость. Команда же — единственная явная точка, где происходит и
переход, и проводка, в одной транзакции.

**Компромиссы.** Нужно дисциплинированно не «дёргать» переходы в обход команд;
обеспечиваем это тем, что переход и проводка атомарны внутри команды.

## ADR-007. REST: переходы статусов — отдельные эндпоинты

**Решение.** Переходы — это действия-подресурсы:
`POST /orders/:id/success` и `POST /orders/:id/cancel`, а не общий
`PATCH /orders/:id` с произвольным `status`.

**Почему.** Каждый переход — отдельная бизнес-команда с своими правилами и
побочными эффектами. Явные эндпоинты безопаснее (нельзя выставить любой статус
напрямую), самодокументируемы и точно ложатся на CQRS-команды и на
`Idempotency-Key`.

**Компромиссы.** Чуть больше маршрутов, чем у «универсального update». Для
платежей явность и безопасность важнее.

## ADR-008. Инфраструктура и стек

- **Rails 8.1 API-only** — фронтенд не нужен; меньше поверхность, чище код.
- **PostgreSQL 18** — ACID, строчные блокировки, уникальные/частичные индексы и
  CHECK-констрейнты — всё, на чём держатся ADR-002/003/004.
- **Docker Compose + Makefile** — `make up` поднимает весь стек одной командой;
  проверяющему не нужно ставить Ruby/Postgres локально.
- **`panko_serializer`** — быстрая и явная сериализация (требование ТЗ); схема
  ответа задаётся кодом, а не магией.
- **RSpec, только request-спеки** — тестируем поведение через публичный контракт
  API (вход → статус/тело), а не приватные детали; это даёт широкое покрытие
  бизнес-сценариев и устойчивость к рефакторингу.
- **`isolator`** — в тестах ловит небезопасные не-атомарные операции (например,
  внешний вызов внутри открытой БД-транзакции) — профильная страховка для
  платежей.

## ADR-009. Направление движения средств и достаточность средств

**Решение.** Счёт — это **кошелёк** юзера. **Успех заказа дебетует счёт**
(баланс уменьшается на сумму заказа), **отмена успешного заказа сторнирует**
(возврат: баланс растёт обратно). При успехе под локом проверяем
**достаточность средств**: если `balance_cents < amount_cents` — операцию
отклоняем (`422`), заказ остаётся `created`, проводки нет.

**Почему.** Это естественная семантика кошелька: нельзя потратить больше, чем
есть. Проверку делаем внутри той же транзакции под `SELECT … FOR UPDATE`, чтобы
два параллельных списания не «пробили» баланс в минус (TOCTOU/гонка). Возврат
(reversal) всегда разрешён и баланс только увеличивает.

**Подкрепляем на уровне БД.** `CHECK (balance_cents >= 0)` — инвариант хранилища:
даже при баге в коде БД не даст уйти в минус (defense-in-depth). Знак движения
храним прямо в `transactions.amount_cents` (settlement `< 0`, reversal `> 0`),
поэтому `balance_cents == SUM(transactions.amount_cents)` — простой проверяемый
инвариант.
